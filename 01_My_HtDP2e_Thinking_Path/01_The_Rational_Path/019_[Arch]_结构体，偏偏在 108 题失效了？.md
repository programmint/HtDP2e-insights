## 4 次迭代，方才发现真正数据定义

108 题为什么不使用结构体，而仅仅只用字符串和数值型，是从错误中迭代出来，下面一一复述其过程。

<br>

### 第1次迭代：读错题目

108 题，上来我就误解了题意：


>
><br>
>
>其实这个题目，很简单。红绿灯。
>
>1、默认状态：红灯。
2、按空格后，变成绿灯，同时显示倒计时数字（9->0） **<<< 读错题目了**
3、恢复为默认红灯状态。
>
>考虑到我比较喜欢加戏，例如，把绿灯和倒计时的数字并排，其实题目里已经讲了，显示完了绿灯，就显示倒计时的数字。
>
>显然这个题目并不复杂。
>
>作为灯的状态，其实就两种，要么是红灯，要么是绿灯，不用想，cond 来解决。
>
>绿灯之后，要显示数字，这里要把时钟滴答的频率改为1秒钟，从 9 开始，时钟每滴答一次，在当前的数字基础之上，就减少一秒，等于 0时，切换为红灯状态。
>
>这个题目的难点，就是展示倒计时这里，时钟的频率要修改一次。
> <br>
>

<br>

>
><br>
>
>我已经知道这个题目怎么样来做了。
>
>给三个状态命名，常量。
>
>big-bang 只需要3个子程序，on-tick，on-key，on-draw
>
>on-tick 的频率是 1 秒，不管时钟怎么样滴答，如果是红灯状态，依然返回红灯状态。什么都不做。
>
>如果是绿灯状态，1秒钟后，就返回倒计时状态。**<<< 还是读错题目**
>
>如果是倒计时状态，就倒计时10秒钟，返回倒计时状态。超过了10秒钟，就返回红灯状态。
>
>on-key，去检验当前的状态，如果是红灯状态，有空格行为，就返回绿灯状态，否则一律返回原有状态。
>

<br>

**开始，我就把题目读错了，我一直都以为绿灯只持续 1 秒钟。**

顺着错误思路，很快就把部分代码写了出来，丢给 Ai： 


```racket
;; === 状态更新函数 ===

(define (update-state state)
    (cond
        [(string=? state DEFAULT-STATE) "red"]
        [(string=? state GREEN-STATE) "countdown"]
        [(string=? state LAST-COUNTDOWN-STATE) "red"]))

```
<br>

### 第 2次迭代：使用结构体
Ai 反问我，倒计时去哪里了？

我也一脸懵，对啊，倒计时去哪里了？

这时我意识到，需要使用结构体了，马上修改思路：

><br>
>红灯、绿灯，都是字符串，只有倒计时的状态，才是结构体，这个结构体，是状态和具体的数字在一起
><br>
><br>

<Br>

Ai 根据我的思路，给了结构体的示例代码
```racket
;; 倒计时结构体：包含状态标签和剩余秒数
(define-struct countdown [tag seconds])
;; tag: 字符串，表示当前阶段，例如 "countdown"
;; seconds: 数字，表示剩余秒数，范围 9 到 0
```
<br>
为什么要使用结构体？

因为要一次传递多个信息，内置的数据类型，无法完成这一点，只能使用结构体了。

<br>

### 第 3 次迭代：签名+语法出错

#### 3.1、签名有问题

><br>
>
>签名有问题，假如倒计时的数字，我规定为文本型，那么状态更新函数，还是 string-> string ，但开始使用结构体时，就是另外的数据类型了，也就是说状态更新函数，有可能使用字符串，也有可能使用结构体，这个签名该怎么样写？
><br>

我这问题，**Ai 评价为：架构的崩溃，往往是从函数签名没法写开始的** (Ai 这文句明显不通顺啊！)

怎么样解决这个问题？也简单，继续使用结构体：

><br>
>
>是的，这个题目，还得用到结构体，这个结构体包含了两种数据格式，文本，文本，另外一个结构体，也就是状态转换的3种形态，（make-light “red” “green” （make-countdown  “countdown” 9））
>
><br>

<br>

Ai 提醒我，红灯、绿灯、倒计时是互斥状态，不能放在一起。

我马上意识到：

><br>
>
>因为3种状态是互斥的，而且三种状态还不一致，但程序又需要这三种状态，其实到了这里，我就应该认识到，这3种状态构成一个大的状态，也就是3种数据定义，构成一个大的数据定义，这个数据定义的方式，只能是数据联合，也就是条目。
>
><br>


第 3 次迭代数据定义之前，我去 github 上把别人代码扒下来,运行，看到别人的绿灯居然持续那么久时间，我还傻乎乎笑别人乱写代码，可也十分好奇，为什么这样子写？

又去看原题目，猛惊醒，我居然误读了题目，绿灯需要持续10秒钟，这一条件直接导致了数据定义的变化。

很快，数据定义写了出来：

```racket
;; === LightState 数据定义 ===

;; LightState 是三者之一:
;; -- “red”：默认状态
;; -- “green”：绿灯状态
;; -- Number：绿灯持续状态[10,19]，及倒计时的数字，[0,9]

;; === 状态更新函数 ===
;; LightState -> LightState
(define (update-state state)
(cond
[(string=? state "red") "red"]
[(number? state) (update-timer state)]))

```

<br>

#### 3.2、语法出问题
可惜，我代码写错了，Ai 给我指出来：

如果 state = 15 ，数据传递到 ：
```
[(string=? state "red") "red"]
```
程序直接报错。

再接着改：

```racket
;; === 状态更新函数 ===
;; LightState -> LightState
(define (update-state state)
    (cond
        [(string? state) "red"]
        [(number? state) (update-timer state)]))
```


<br>
同时把主程序加上防御手段：

```racket
;; === 主程序 ===

;; LightState -> LightState
(define (pedestrian-traffic-light state)
  (cond
    [(valid-lightstate? state) (big-bang state  <-- 验证是不是有效数据？
                                 [on-tick update-state TICK-INTERVAL]
                                 [on-key handle-key]
                                 [on-draw render-traffic-light])]
    [else (error "LightState 要求输入正确的数据格式")]))
```
好了，终于没有什么问题了。
<br>

### 第 4 次迭代：终于发现问题
我的主函数的子函数，第一个就是状态函数，转机就在这里。

我构思状态更新函数，迅速发现一问题：

><br>
>
>我有个问题，其实绿灯维持的 10 秒钟，其实是从 19 开始，然后到了 9 就开始进入倒计时状态，其实这两个阶段，是一个阶段，呈现不同的图像，那是渲染函数的事情，也就是说，绿灯时的状态函数，就是从 19 开始，一直倒数到 0，然后到了0，再变为红色状态。
><br>

<br>

原来 **108 题，程序使用的数据类型，只有两种：字符串、数值。**

**108题使用结构体，不能说是错误的思路，而是一麻烦的思路。**

不停引导你犯错，htdp2e 老习惯了。

马上修改数据定义：

```racket 
;; === LightState 数据定义 ===

;; LightState 是三者之一:
;; -- “red”:默认状态
;; -- Number:绿灯状态，绿灯持续状态[10,19]，及倒计时的数字，[0,9]
```

<br>

## 结构体不适合吗？

不能这样子讲，说穿了，还是数据定义的问题。

[详解数据定义 —— 107 题里的认知演变（2）](./018_[Method]_详解数据定义%20—%20107%20题里的认知演变(2).md) 一文里把数据定义的作用总结为3方面：

><br>
>
>第一步：信息 → 数据
人把意图告知计算机。
>
>**第二步：数据 → 数据**  
计算机承担计算的重任。
>
>第三步：数据 → 信息
人来验收计算成果。
>
><br>

我所仰仗的，正是第二步，我喜欢称之为：**数据定义利于计算。**

**使用结构体，麻烦。** 需要不但拆开、组合结构体，计算起来麻烦。

**使用字符串和数值型，简单得多。**

<br>

所以，108 题里，我的数据定义，放心大胆定义为：

```racket 
;; === LightState 数据定义 ===

;; LightState 是三者之一:
;; -- “red”:默认状态
;; -- Number:绿灯状态，绿灯持续状态[10,19]，及倒计时的数字，[0,9]
```

<br>

108 题的代码里：

```racket 
;; Number -> String
;; 倒计时数字颜色，奇数桔色，偶数为绿色
(define (countdown-number-color state)
  (cond
    [(= (modulo state 2) 1)"orange"]
    [else "green"]))
```
[else "green"] 语句也是为了利于计算，舍弃 [(= (modulo state 2) 0) "green"] 的校验，直接使用 [else "green"]，避免了 CPU 去做无用功。
