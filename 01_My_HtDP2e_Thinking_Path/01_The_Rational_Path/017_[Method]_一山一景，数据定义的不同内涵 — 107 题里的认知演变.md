
# 一山一景，数据定义的不同内涵 —— 107 题里的认知演变

## 1、数据定义的内涵

### 1.1、数据签名
1-4 章的函数签名，使用的都是内置数据，算不上严格的数据定义。

<br>

### 1.2、定义结构体
到了第 5 章的结构体编程，这才有了严格意义上的数据定义。

原文如下：

> 5.6 Programming with Structures
**Proper programming calls for data definitions.** With the introduction of structure type definitions, data definitions become interesting. Remember that a data definition provides a way of representing information into data and interpreting that data as information.
>

**5.6节的原文，起始一句：严谨的编程讲究数据定义。**

后面紧跟的一语，则是非常关键，直接解决了数据定义的作用。

Remember that a data definition provides a way of representing information into data and interpreting that data as information.

这话可以翻译为：

**请记住，数据定义的作用，就是把信息转化为数据，再把数据解读为信息。**

由此而知，数据定义的内涵会不停扩展，但所起的作用则是明确了：

信息-->数据

数据-->信息

二者缺一不可。

而这一句话，其实我多多少少有点忽视了。

<br>

### 1.3、数据定义与程序设计
htdp2e 的第3章《程序设计方法》（3 How to Design Programs）

不得不吐槽：书名居然和一章节的名字重合，有点奇怪。

>原文：
>
>Given the central role of information and data, program design must start with the connection between them. Specifically, we, the programmers, must decide how to use our chosen programming language to represent the relevant pieces of information as data and how we should interpret data as information. Figure 15 explains this idea with an abstract diagram.
>
>可以翻译为：
>
>信息与数据处于核心地位，程序设计须始于二者之关联。具体而言，程序员须决定：如何用编程语言将信息编码为数据，又如何将数据解码为信息。
>

程序设计将信息与数据关联起来，如何把二者关联起来，固定数据章节里，这一重任由数据定义来完成。

<br>

## 2、需求与数据定义的矛盾

### 2.1、需求先行

htdp2e 的习题有一特点：上限可以无限高。

所以，我把 107 题写成了一小型游戏。构思过程中，之前的一认知，和 106、107 题动手打了起来，直接发生了碰撞。

之前的认知，写了一文章：**[001_[Method]_反思设计诀窍](./001_[Method]_反思设计诀窍.md)** ，留为笔记。

此文主题：

**繁杂的需求，决定着怎么样的数据定义。**

程序设计过程中，难道不是：函数目的声明在前，函数签名靠后吗？

美其名曰：修正 htdp2e 之不足。

**写完这文章，感觉自己太厉害了，人开始有点飘。**

后续题目，函数目的声明，一律在前，函数签名靠后站。

<br>

### 2.2、开始撞墙

直至 106 题，特别是 107 题，写函数的过程中，心安理得开始胡编乱造。

例如，106 题：

~~~
;; VAnimal是两者之一：
;; -- VCat
;; -- VCham
~~~

<br>

写到要更新猫或者变色龙的状态，顺手就造了一个新函数：
~~~
;; 实时更新猫或变色龙的各个状态
;; WorldState -> WorldState
(define (update-state state)
  (make-animal         ;; <---- 致命错误在这里 ！
   (next-x state)
   (next-happiness state)
   (next-direction state)))

~~~

当时的思路理直气壮，要更新状态，重新定义一个新函数就好了，

函数头：(update-state state)

函数体：反正都是动物，那就 make-animal 就好了。

Ai 无情拷问上来，make-animal 是什么，从哪儿冒出来？

我心想：Ai 你瞎啊，看不清吗，make-animal 啊？

**之前傲慢的心态，导致犯错，愣是不自知，一再重复错误。** 

正确的思路，该这样思考：
~~~
;; VAnimal是两者之一：
;; -- VCat
;; -- VCham
~~~

<br>

数据定义是"两者之一（Union）"，代码只能长成下面“模板”的形状，根本不用想，直接使用 cond 来完成，即：

~~~
(define (update-state state)
  (cond
    [(vcat? state) (...处理猫的逻辑...)]
    [(vcham? state) (...处理变色龙的逻辑...)]))
~~~

表面上，我对模板的认知，比较肤浅，实则另外一回事。

写完了那文章，我注重每一个函数的任务是什么，不同函数之间该怎么样来配合，共同完成某一任务。

好事：我悄悄摸到了系统设计的领域。

坏事：忽视了基础的锤炼，不知不觉中忽略了数据定义的作用。

函数的本质工作，仅一项：计算。利用数据，完成计算。

**函数只能使用数据定义，不能硬造一数据定义。**

<br>

### 2.3、激烈撞车

106 题给 Ai 拷问多了，偷偷把函数签名的位置，挪至第 1 位，初步感受到了数据定义的意义。

107 题，我自己不断加戏，题目越来越复杂，Ai 不停反问：看看你的函数签名，我才慢慢看清了思路中的裂缝。

写 [001_[Method]_反思设计诀窍](./001_[Method]_反思设计诀窍.md) 时，刚学 htpd2e，还用原有的思维模式，看待新事物，觉得 htdp2e 有缺陷，不足为奇。

解 107 题过程中，给狠狠收拾了一通：
~~~

- 你照着设计配方，定义好了 VCat、VCham、Zoo。然后你发现，光有数据结构不够，你需要一个格斗游戏的状态栏。

- 你开始设计状态栏。然后你发现，光有布局不够，你需要管理几十个常量。

- 你开始管理常量。然后你发现，光有分组不够，你需要确保依赖顺序不出错。

- 你开始排依赖顺序。然后你发现，光有顺序不够，你需要让间距可调、对齐方向可配、血条向中心缩减。

- 你开始调整间距和对齐。然后你发现，每改一个地方，别的地方就崩了。

 <注：这段文字，由 Ai 协助整理>
 <Ai 总结得更符合时间顺序，我自己的思维跳来跳去，未必总结得那么符合时间线>

~~~

一句话总结，需求先行，肯定没错。

可是写函数的过程，需求则是一概置之不理，一甩手掌柜，其实这些问题，都要依靠数据定义来完成。

<br>

### 2.4、二者合一
我把 107 题写成小型游戏，程序确实变复杂了，也确实看清了设计的过程。

**第1、构思阶段**
用自然语言描述意图，该有什么模块，函数怎么样分组，常量怎么样分组，利用自然语言快速搭建起来。

例如，我是如下的方式，来规划各个模块

~~~
;; ------------------------------------------
;; 游戏场景图像

(define SCENE-WIDTH 300)
(define SCENE-HEIGHT 300) ; 场景唯一高（多处用）
(define SCENE (empty-scene SCENE-WIDTH SCENE-HEIGHT))

;; ------------------------------------------
;; 快乐指数图像

;; 快乐面板宽

;; 快乐面板图像

;; 快乐指数条宽

;; ------------------------------------------
;; 猫图像

;; 行走猫

;; 睡觉猫

;; ------------------------------------------
;; 变色龙图像

;; 开心变绿色

;; 生气变色龙
~~~

这个阶段，时刻盯着程序的意图，注重各个模块配合得当，根本不考虑细节。

（注：其实从这里也看出来，自然语言模糊，Ai 编程就会把这个问题给放大了，所写未必乃所需。）

<br>

**第2、书写阶段**

这一阶段，开始写代码，数据定义就必须先于函数存在，函数使用的数据，都必须已经被定义过。

**第1、该使用什么数据？（要预先定义好）** 

**第2、后续的计算过程要怎么样兼顾？（定义是为了计算）**

**第3、不同的数据之间怎么样配合？（函数之间怎么样配合）**

写的过程，有可能局部调整数据定义，例如我写的 107 题，有的可能是翻天覆地重新规划数据定义，暂时还没有碰到。

### 2.5 二者阶段不同，并无矛盾

总结：二者并无矛盾。

需求决定了数据定义，数据定义决定函数形式。

二者出场的时间不同，共乘一船，非两种对立的思路。
